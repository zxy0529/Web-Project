
## **1. 引言**
### **1.1 编写目的**
本需求规格说明书旨在明确图书馆自习室预约系统的功能和非功能性需求，为开发团队提供清晰的指导，确保系统的开发符合学校图书馆的管理需求及用户的使用需求。同时，本文件将作为项目实施过程中需求确认、开发、测试和后续维护的重要依据。
### **1.2 项目背景**
当前，许多高校图书馆面临座位管理混乱的问题，如占座、资源浪费、找座困难等，尤其在考试季更加突出。传统的人工管理模式效率低，难以满足日益增长的使用需求。因此，开发一套智能化的自习室预约系统，可以有效提升图书馆资源的利用率，提供更加公平、便捷的座位管理方式，提高师生的学习体验。

本系统的主要应用场景包括：
1. **学生**：在线预约图书馆座位，避免寻找座位的困扰。
2. **教师**：预定座位用于学术研究、辅导学生等活动。
3. **管理员**：管理预约数据、调整座位规则，确保公平高效的使用。
### **1.3 定义**

|**术语/缩写**|**定义**|
|---|---|
|预约|用户通过系统预定一个特定时间段的座位。|
|签到|用户到达自习室后，通过扫码或人脸识别确认占座。|
|违约|用户预约后未签到或未按规则使用座位，可能导致惩罚。|
|高峰期|座位需求量较大的时间段，如考试周。|

### **2 总体描述**
### **2.2 需求概述**
图书馆自习室预约系统旨在为师生提供便捷、高效的自习座位管理服务。该系统允许用户通过网页或移动端预约自习座位，支持查询座位使用情况、个性化预约（如选择带插座、靠窗等座位）、取消预约及违约处理等功能。同时，管理员可以管理座位配置、审核特殊申请、监控使用数据，并优化资源调配。系统的目标是提升图书馆空间利用率，减少占座及空置率，打造公平、高效的学习环境。
### **2.2 用户类和特性**

|**用户类型**|**功能需求**|
|---|---|
|普通学生|注册登录、预约座位、签到、取消预约、查询座位状态|
|教师|预约座位、长时间预约权限|
|管理员|管理座位资源、设置规则、监控预约数据、处理违规行为|
###  **2.3 运行环境**
##### 硬件环境
- 处理器: 12th Gen Intel(R) Core(TM) i5-12500H 2.50 GHz  
- 机带 RAM: 16.0 GB (15.7 GB 可用)  
- 系统类型: 64 位操作系统, 基于 x64 的处理器  
##### 软件环境 
1. 操作系统: Windows 11 
2. 开发工具
- IDE: Visual Studio Code、IntelliJ IDEA、Eclipse 等，用于编写、调试和测试代码。
- 版本控制: Git，用于代码的版本管理，可结合 GitHub、GitLab 等平台进行团队协作。
3. 前端开发
- 编程语言: JavaScript，结合 HTML5、CSS3 进行页面构建和样式设计。
- 前端框架: React.js 或 Vue.js，用于构建单页面应用（SPA），提高用户体验和页面加载速度。
- UI 设计工具: Sketch、Figma 等，用于设计用户界面和交互原型。
4. 后端开发 (Backend Development)
- 编程语言: Java（Spring Boot 框架或 SSM 框架）或 Node.js（Express 框架）。
- 数据库: MySQL 或 PostgreSQL，用于存储系统的结构化数据，如用户信息、订单信息、菜品信息等。
- 数据库管理工具: Navicat、DBeaver 等，用于数据库的创建、管理、备份和恢复。
5. 测试环境 (Testing Environment)
- 单元测试: JUnit（Java）或 Jest（JavaScript），用于测试后端和前端代码的逻辑正确性。
- 集成测试: 使用 Spring Boot Test（Java）或 Mocha（JavaScript）等框架，测试后端与前端、数据库等组件的集成情况。
- 性能测试: 使用 JMeter 或 Gatling 等工具，模拟高并发访问，测试系统的响应时间和吞吐量。
### **2.4 设计和实现上的限制**
1. **系统需支持高并发访问**，尤其在考试周等高峰期。
2. **数据安全性要求高**，确保用户信息和预约记录不被篡改。
3. **设备兼容性**，系统需适配不同设备（PC、手机）。
4. **与现有校园系统集成**，如身份认证系统（统一认证登录）。
## **3. 功能需求**
### **3.1 功能划分**

| **模块名称** | **功能描述**                     |
| -------- | ---------------------------- |
| 用户管理     | 用户注册、登录、身份认证                 |
| 座位预约     | 预约座位、筛选座位、取消预约               |
| 预约提醒     | 提醒用户预约时间、座位信息<br>            |
| 签到验证     | 预约后签到、扫码签到、人脸识别签到            |
| 违约管理     | 预约后未签到处罚、预约取消限制              |
| 临时离开模式   | 用户可以设置临时离开座位，并在返回时恢复预约状态<br> |
| 座位管理     | 管理员添加/删除/调整座位、设定规则           |
| 统计分析     | 预约数据分析、使用率统计                 |


### **3.2 功能详细描述**
#### **3.2.1 用户管理**
**输入**：用户注册信息（学号/工号、姓名、邮箱、密码）  
**处理**：校验用户身份，创建账户，绑定学校统一身份认证  
**输出**：用户登录后可访问个人主页
#### **3.2.2 座位预约**
**输入**：用户选择日期、时间、座位类型  
**处理**：检查座位是否可用，完成预约，生成预约记录  
**输出**：预约成功通知，用户可查看预约信息

#### **3.2.3 预约提醒**
**输入**：用户预约成功后的信息（预约时间、座位号、用户联系方式）  
**处理**：
1. 预约成功后，系统向用户发送 **短信、邮件或微信/APP推送** 预约确认通知。
2. 若用户未按时签到，发送 **未签到提醒**，提示用户即将违约。  
**输出**：通知发送成功，用户收到提醒信息。
#### **3.2.4 签到验证**
**输入**：用户扫描二维码 / 通过账号签到  
**处理**：
1. 若预约成功且在允许签到时间内，系统更新座位状态为“使用中”。
2. 若用户未签到，系统在截止时间前发送“未签到提醒”。  
**输出**：用户签到成功，座位状态更新。
#### **3.2.5 违约管理**
**输入**：用户未在规定时间内签到或提前离开  
**处理**：
-  若用户 **连续 3 次** 违约（未签到 / 提前离开），系统**暂停预约权限 7 天**。
- 若 **累计 5 次** 违约，系统**暂停预约 30 天**。
- 用户可提交申诉，由管理员审核是否解除黑名单。  
**输出**：
- 用户信用分更新，黑名单用户将无法预约。
- 用户可在个人中心查看信用记录。
#### **3.2.6 临时离开模式**
**输入**：用户签到后的座位状态变更请求（临时离开）  
**处理**：
1. 用户签到后，可点击 **“临时离开”** 按钮，系统标记座位状态为“暂离”。
2. 设定离开 **最大时长**（如 30 分钟，管理员可调整）。
3. **超时未归**：
    - **提醒用户**：临时离开剩余 5 分钟时，系统发送提醒。
    - **自动释放**：超时后，座位自动释放，用户被记录违约。  
**输出**：
- 用户座位状态更新（离开/返回）。
- 超时未归的用户被记录违约。
#### **3.2.7 座位管理**（管理员权限）

**输入**：管理员设置座位分布、开放时间、预约规则  
**处理**：系统应用设置，调整座位状态  
**输出**：用户可在前端看到最新座位信息

#### **3.2.8 统计分析**（管理员权限）

**输入**：系统日志、用户预约记录  
**处理**：统计座位利用率、违约率等数据  
**输出**：生成数据报表，管理员优化管理策略

### **3.3 优先级和约束**

| **功能** | **优先级** | **约束条件**                |
| ------ | ------- | ----------------------- |
| 用户管理   | 高       | 必须与学校统一身份认证系统对接         |
| 座位预约   | 高       | 预约时间和规则需符合管理员设定         |
| 签到验证   | 高       | 需要扫码设备或人脸识别设备支持         |
| 预约提醒   | 中       | 必须通过系统自动提醒用户预约时间和状态     |
| 违约管理   | 中       | 违约规则需符合学校管理规定           |
| 座位管理   | 中       | 仅管理员有权限管理座位             |
| 临时离开模式 | 中       | 用户需手动设置临时离开，且系统自动恢复预约状态 |
| 统计分析   | 低       | 仅管理员可查看，定期导出报表          |
### **3.4 非功能需求**
#### **3.4.1 可用性**
- **易用性**: 系统应具有简洁、直观的用户界面，确保用户在第一次使用时能够快速上手。界面设计应符合现代化的审美标准，操作流程应清晰明了，用户可以轻松完成预约、签到、取消等操作。
- **可学习性**: 系统应具有易于理解的帮助文档或新用户引导功能，使新用户能够迅速掌握如何使用系统的各项功能。还应提供清晰的错误提示和帮助信息，便于用户解决问题。
- **可访问性**: 系统应符合常见的无障碍设计规范，支持不同需求用户的使用，确保所有用户都能顺利使用系统。
#### **3.4.2 可维护性**
- **模块化设计**: 系统应采用模块化设计，确保每个模块功能独立且易于维护。功能模块如预约管理、签到验证、违约管理等应清晰划分，便于后期进行功能扩展或维护。
- **文档完备性**: 系统应提供完整的技术文档，包括系统架构、API接口文档、数据库设计文档、代码注释等，确保开发和维护人员可以迅速理解和修改系统。
- **可扩展性**: 系统设计时应考虑到未来功能扩展的可能性，新的功能模块能够无缝集成到现有系统中，避免过多的修改现有代码。
#### **3.4.3 可移植性**
- **跨平台支持**: 系统应支持常见操作系统（如 Windows、macOS、Linux）上的运行，特别是作为Web应用时应支持跨平台浏览器访问。
- **环境适应性**: 系统在不同网络环境中应具有良好的适应性，保证在低带宽或高延迟环境下仍能正常工作。
#### **3.4.4 兼容性**
- **操作系统**: 系统应兼容Windows、macOS和Linux操作系统，并确保在这些平台上的兼容性。
- **浏览器**: 系统应支持主流浏览器，包括Google Chrome、Mozilla Firefox、Microsoft Edge和Safari等，以确保用户能够通过不同浏览器访问系统。
- **硬件**: 系统应支持常见的硬件设备，如扫码设备、计算机、平板、移动设备等。对于特殊硬件（如人脸识别设备）应提供明确的硬件要求。
### **3.5 其他需求**
#### **3.5.1 数据管理和存储**
- **数据存储**: 系统应采用可靠的数据库存储所有用户信息、预约数据、签到记录等。数据库应定期备份，并确保数据的安全性和完整性。
- **备份策略**: 系统应定期进行数据备份（例如，每日或每周自动备份），并存储多个备份副本。备份文件应加密存储，防止数据丢失或泄露。
- **数据完整性**: 系统应使用事务管理和数据验证机制确保数据一致性和完整性。对用户的所有操作进行记录，确保在系统出现故障时能够恢复到最后一致的状态。
#### **3.5.2 法律和合规性**
- **法律法规**: 系统应遵守相关的法律法规，例如数据保护法（如《中华人民共和国个人信息保护法》），确保用户个人信息的隐私性和安全性。用户数据应仅限于预定的用途，并严格限制访问权限。
- **合规性要求**: 系统应遵循与学校和相关教育机构的合规性要求，确保符合学校管理规定，并能适应相关教育政策变化。
- **标准化**: 系统应符合行业标准，如信息安全标准、数据保护标准等，确保系统的长期可用性和法律合规性。
